# Звіт: масштабування backend системи MediDispense

## 1. Опис стратегії масштабування

У лабораторній роботі для системи MediDispense було реалізовано горизонтальне масштабування backend. Суть цього підходу полягає в тому, що один і той самий ASP.NET Core Web API запускається у кількох однакових екземплярах, а вхідні HTTP-запити розподіляються між ними.

Backend було розгорнуто у Kubernetes як `Deployment`. Кількість екземплярів backend змінюється через кількість pod-ів. При збільшенні навантаження можна збільшити кількість pod-ів, а при зменшенні навантаження - зменшити її. Запити користувачів надходять не напряму до конкретного pod-а, а через Kubernetes `Service`, який виконує роль балансувальника навантаження.

Також було додано автоматичне масштабування через Kubernetes `HorizontalPodAutoscaler`. HPA відстежує CPU-навантаження backend pod-ів і може автоматично змінювати кількість екземплярів API в межах від 1 до 5 pod-ів. Це дозволяє системі реагувати на зміну навантаження без ручного втручання.

База даних PostgreSQL винесена в окремий pod і використовується всіма екземплярами backend. Таким чином, API pod-и залишаються stateless: вони не зберігають критичний стан локально, тому будь-який pod може обробити запит користувача.

## 2. Опис тестового середовища

Для виконання роботи використовувався проєкт MediDispense - backend система для керування процесами видачі медикаментів. Backend реалізований на C# з використанням ASP.NET Core Web API (.NET 9). Для збереження даних використовується PostgreSQL.

Інфраструктура була розгорнута у локальному Kubernetes кластері Docker Desktop з provider `kind`. Для балансування навантаження використовувався Kubernetes `Service` типу `LoadBalancer`. Для навантажувального тестування використовувався Locust, який також запускався у Kubernetes як окремий pod.

Для backend pod-ів були задані обмеження ресурсів: кожен pod запитує 0.1 CPU core і 128 MiB RAM, а максимальне використання одного pod-а обмежене 0.5 CPU core і 512 MiB RAM. Такі обмеження потрібні для того, щоб у локальному середовищі було видно вплив масштабування: при збільшенні кількості pod-ів система отримує більше сумарних ресурсів для обробки запитів.

Для автоматичного масштабування було встановлено `metrics-server`, оскільки саме він надає Kubernetes метрики CPU, необхідні для роботи HPA.

## 3. Технічні рішення для масштабування

Масштабування стало можливим завдяки тому, що backend API не прив'язаний до конкретного екземпляра сервера. Дані зберігаються у PostgreSQL, а авторизація використовує JWT token-и з однаковими secret keys для всіх pod-ів. Це означає, що token, створений або перевірений одним pod-ом, буде валідним і для інших pod-ів.

Kubernetes `Deployment` відповідає за створення потрібної кількості backend pod-ів. Kubernetes `Service` знаходить усі pod-и з відповідним label і розподіляє між ними запити. Для підтвердження роботи балансування було додано endpoint `/api/diagnostics/instance`, який повертає назву pod-а, що обробив запит. При кількох pod-ах повторні запити повертають різні назви екземплярів, що підтверджує розподіл навантаження між ними.

Для створення навантаження було додано endpoint `/api/diagnostics/work`. Він виконує CPU-bound роботу і дозволяє краще показати вплив масштабування backend, оскільки при збільшенні кількості pod-ів зростає сумарна кількість CPU-ресурсів, доступних для API.

Конфігурації Docker, Kubernetes, HPA і Locust були винесені в окремі файли репозиторію. Це дозволяє відтворити середовище на іншій машині та завантажити всі налаштування на GitHub разом із кодом проєкту.

## 4. Аналіз вузьких місць

Основним вузьким місцем при навантаженні для backend API є CPU. Це особливо помітно на endpoint-ах, які виконують обчислення, серіалізацію JSON або іншу роботу на стороні сервера. Саме тому для тестування використовувався CPU-bound endpoint `/api/diagnostics/work`.

Другим потенційним вузьким місцем є PostgreSQL. У цій роботі масштабувався саме backend, а база даних залишалася єдиною. Це означає, що при збільшенні кількості API pod-ів зростає кількість потенційних одночасних підключень до бази даних. Якщо запити до бази стануть важкими або їх буде дуже багато, bottleneck може перейти з backend API на PostgreSQL.

Ще одним обмеженням є локальне середовище Docker Desktop. Усі pod-и працюють на одному комп'ютері, тому приріст продуктивності обмежується ресурсами цієї машини, налаштуваннями Docker Desktop і можливостями генератора навантаження Locust. Через це в локальному тесті RPS може не зростати пропорційно кількості pod-ів, навіть якщо latency покращується.

## 5. Навантажувальне тестування Locust

Для перевірки роботи системи під навантаженням використовувався Locust. Він імітував одночасну роботу користувачів і вимірював кількість запитів за секунду, час відповіді та кількість помилок.

Сценарій Locust складався з кількох типів запитів. Спочатку користувач авторизується через `/api/auth/login`, після чого викликаються реальні endpoint-и системи, зокрема `/api/medications`. Додатково викликались diagnostic endpoint-и `/api/diagnostics/instance` та `/api/diagnostics/work`. Основним endpoint-ом для оцінки впливу масштабування був `/api/diagnostics/work`, оскільки він створює CPU-навантаження на backend.

Тестування проводилось з однаковими параметрами для різної кількості backend pod-ів: 200 користувачів, spawn rate 40 користувачів за секунду, host `http://medidispense-api:8080`. Перед порівнянням ручного масштабування HPA було тимчасово вимкнено, щоб кількість pod-ів залишалася фіксованою протягом тесту.

Після цього було окремо перевірено автоматичне масштабування. Для цього HPA було увімкнено з параметрами: мінімум 1 pod, максимум 5 pod-ів, цільове CPU-навантаження 50%. Під час навантаження HPA показував збільшення CPU utilization і автоматично додавав нові backend pod-и.

## 6. Результати тестування

Основне порівняння виконувалось між одним backend pod-ом і п'ятьма backend pod-ами. Результати Locust наведені нижче.

| Режим | Кількість API pod-ів | Users | Spawn rate | Average RPS | Average response time | Max response time | Failures |
| --- | ---: | ---: | ---: | ---: | ---: | ---: | ---: |
| Manual scaling | 1 | 200 | 40 | 452.4 | 41.67 ms | 18143 ms | 0% |
| Manual scaling | 5 | 200 | 40 | 604.3 | 11.81 ms | 2795 ms | 0% |

При одному pod-і система обробила 17005 запитів, середній RPS становив 452.4, середній час відповіді - 41.67 ms, а максимальний час відповіді досягав 18143 ms. Помилок під час тесту не було.

При п'яти pod-ах система обробила 32558 запитів, середній RPS становив 604.3, середній час відповіді зменшився до 11.81 ms, а максимальний час відповіді зменшився до 2795 ms. Помилок також не було.

Детальні результати по endpoint-ах:

| Кількість pod-ів | Endpoint | Requests | Average response time | Max response time | Current RPS | Fails |
| ---: | --- | ---: | ---: | ---: | ---: | ---: |
| 1 | `/api/diagnostics/instance` | 2424 | 35.58 ms | 7111 ms | 65.9 | 0 |
| 1 | `/api/diagnostics/work` | 9812 | 37.8 ms | 7238 ms | 261.6 | 0 |
| 1 | `/api/medications` | 4769 | 52.71 ms | 18143 ms | 124.9 | 0 |
| 5 | `/api/diagnostics/instance` | 4611 | 8.85 ms | 1506 ms | 84.9 | 0 |
| 5 | `/api/diagnostics/work` | 18648 | 10.29 ms | 1503 ms | 344.1 | 0 |
| 5 | `/api/medications` | 9299 | 16.34 ms | 2795 ms | 175.3 | 0 |

За результатами тестування збільшення кількості backend pod-ів з 1 до 5 підвищило середній RPS з 452.4 до 604.3. Це означає, що при більшому навантаженні горизонтальне масштабування дозволило системі обробляти більше запитів за секунду.

Також масштабування суттєво покращило latency. Average response time зменшився з 41.67 ms до 11.81 ms, а максимальний час відповіді зменшився з 18143 ms до 2795 ms. Failures в обох тестах залишилися на рівні 0%. Це показує, що кілька backend pod-ів не тільки збільшують пропускну здатність, але й роблять відповіді системи стабільнішими під навантаженням.

Під час окремого тесту HPA CPU-навантаження перевищувало цільове значення 50%, після чого Kubernetes автоматично збільшував кількість backend pod-ів. Це підтверджує, що система підтримує не лише ручне масштабування, але й автоматичне масштабування залежно від поточного навантаження.

## 7. Висновок

У роботі було показано горизонтальне масштабування backend системи MediDispense у Kubernetes. Backend було розгорнуто у вигляді кількох однакових pod-ів, а Kubernetes Service виконував роль балансувальника навантаження між ними. Коректність балансування підтверджена endpoint-ом `/api/diagnostics/instance`, який повертав різні назви pod-ів при повторних запитах.

Навантажувальне тестування Locust показало, що збільшення кількості backend pod-ів з 1 до 5 значно зменшило середній і максимальний час відповіді без появи помилок. У локальному середовищі RPS майже не змінився, що пояснюється обмеженнями Docker Desktop, Locust і локального Kubernetes. Попри це, результати підтверджують, що горизонтальне масштабування покращує стабільність системи під навантаженням.

Додатково було налаштовано HPA, який автоматично збільшує кількість pod-ів при зростанні CPU-навантаження. Це робить систему більш адаптивною до змін інтенсивності трафіку.

Усі конфігурації, необхідні для відтворення роботи, знаходяться в репозиторії: Dockerfile, Kubernetes manifests, HPA configuration, Locust scenario та документація.
